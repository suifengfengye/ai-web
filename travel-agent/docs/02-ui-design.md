# Travel Agent UI 设计方案

## 1. 设计目标

Travel Agent 的 UI 不是传统聊天页，而是“对话式规划 + 结构化结果 + 可编辑行程”的混合体验。

核心目标：

- 让用户快速输入需求，不被复杂表单劝退。
- 让结果不是一段大文本，而是可浏览、可修改、可保存的旅行方案。
- 让 AI 的工作过程可感知，包括检索中、分析中、改写中。
- 让后续扩展收藏、导出、分享、下单时无需推翻重做。

## 2. 页面结构规划

建议先做 5 个核心页面。

### 2.1 首页 Landing

目标：解释产品价值，降低首次使用门槛。

模块建议：

- 顶部导航：Logo、登录、我的行程。
- Hero 区：一句话价值主张，例如“告诉我去哪、玩几天、预算多少，我来给你一份可执行行程”。
- 快速输入框：目的地、日期、人数、预算、偏好。
- 示例卡片：周末短途、亲子游、Citywalk、出境自由行。
- 能力介绍：天气、知识库、地图、预算、改写。

### 2.2 对话规划页

这是核心页面，建议采用“两栏布局”。

- 左栏：对话区。
- 右栏：结构化行程预览区。

左栏包含：

- 顶部会话信息。
- 消息流。
- 输入框。
- 快捷按钮，例如“更省钱”“少走路”“加美食”“适合老人”“换住宿区域”。

右栏包含：

- 行程概览卡片。
- 每日安排卡片。
- 预算卡片。
- 天气卡片。
- 注意事项卡片。
- 导出 / 收藏 / 分享操作。

### 2.3 行程详情页

目标：展示最终方案，适合保存和分享。

模块建议：

- 行程头图 / 封面。
- 目的地摘要。
- 每日行程时间轴。
- 地图与路线。
- 预算明细。
- 餐饮、住宿、打包清单。
- 版本记录。

### 2.4 我的行程页

目标：管理历史方案。

模块建议：

- 草稿。
- 已收藏。
- 已分享。
- 最近编辑。
- 标签筛选，如亲子、周末、出境、预算型。

### 2.5 管理后台

MVP 可不做完整后台，但建议预留。

核心能力：

- Prompt 管理。
- MCP 配置管理。
- 知识库文档管理。
- 会话检索。
- 失败请求分析。
- 热门目的地统计。

## 3. 关键交互设计

## 3.1 输入交互

建议采用“自然语言输入 + 结构化补充”的方式。

例如用户输入：

```text
帮我做一份重庆 3 天攻略，五一出发，2 个成年人，预算 4000，想轻松一点，多安排夜景和火锅。
```

系统自动提取结构化字段：

- 目的地：重庆
- 天数：3 天
- 日期：五一假期
- 人数：2 成人
- 预算：4000
- 偏好：轻松、夜景、火锅

若信息缺失，则通过追问卡片补充：

- 出发城市
- 交通方式
- 酒店偏好
- 是否带老人 / 小孩

## 3.2 Agent 过程可视化

不要只显示“AI 正在思考”。

建议分阶段展示：

- 正在解析需求。
- 正在查询天气。
- 正在检索旅游知识库。
- 正在补充景点与路线信息。
- 正在生成行程方案。

这类状态可结合流式输出与工具调用日志渲染，增强可信度。

## 3.3 结果呈现

结果不要只有大段文本，建议拆成卡片：

- 概览卡片。
- 每日时间轴卡片。
- 预算卡片。
- 天气与穿搭卡片。
- 餐饮推荐卡片。
- 住宿区域建议卡片。
- 注意事项卡片。

## 3.4 结果修改

用户最常见的需求不是重新生成，而是局部调整。

建议提供：

- 一键改写按钮。
- 对每一天单独重写。
- 替换单个景点。
- 调整节奏为“轻松 / 紧凑”。
- 预算提高 / 压缩。
- 增加亲子 / 老人友好 / 雨天备选。

## 4. 信息架构

建议把核心数据对象设计为：

- Trip：一次完整旅行计划。
- DayPlan：每日计划。
- Activity：单个活动。
- BudgetItem：预算项。
- UserPreference：用户偏好。
- Conversation：会话记录。

前端界面应围绕这些对象设计，而不是围绕一段文本设计。

## 5. 视觉风格建议

建议风格关键词：

- 清爽。
- 轻出行。
- 高可信。
- 卡片化。
- 地图感。

颜色建议：

- 主色：蓝绿系，传达出行、地图、规划感。
- 辅色：暖橙色，用于 CTA、预算提醒、天气提示。
- 中性色：偏浅灰白，保证阅读体验。

风格建议参考：

- Notion 的信息组织感。
- Airbnb 的旅行氛围。
- Linear 的简洁交互。
- Google Travel 的结构化呈现。

## 6. 组件清单

建议优先沉淀以下组件：

- ChatMessage
- PromptInput
- StructuredTripCard
- DayTimeline
- BudgetBreakdown
- WeatherCard
- POICard
- HotelAreaCard
- LoadingSteps
- QuickActionChips
- ShareDialog
- ExportPanel

## 7. 响应式布局

桌面端：

- 左右双栏最合适。
- 左侧对话，右侧结果。

平板端：

- 默认上下布局。
- 支持切换“对话 / 结果”。

移动端：

- 首页保留轻表单。
- 对话和结果切成 Tab。
- 结果页以卡片和时间轴为主。

## 8. UI 如何借助 AI 设计出来

建议采用“设计系统先行 + AI 出草图 + 人工收敛”的方式，而不是完全交给 AI 自动生成。

### 8.1 推荐流程

1. 先写页面结构和模块清单。
2. 用 AI 生成低保真页面描述。
3. 在 Figma 或 v0 / Galileo / Motiff 中出第一版界面。
4. 基于你自己的品牌色、组件规范收敛成设计系统。
5. 再让 AI 辅助生成高保真文案、图标建议、插画方向和前端代码草稿。

### 8.2 可用工具

- v0：适合从文本直接生成 React 界面草图。
- Figma AI：适合扩展页面和组件变体。
- Galileo AI / Motiff：适合快速出视觉方案。
- ChatGPT / Claude：适合生成页面结构、信息架构、交互说明。

### 8.3 建议给 AI 的设计提示词

```text
请为一个中文 AI 旅游助手设计桌面端 Web 界面。
产品特点：对话式生成旅游攻略，右侧显示结构化行程，包含天气、预算、每日行程、餐饮和住宿建议。
风格关键词：清爽、专业、地图感、现代、轻旅行。
页面包含：首页、对话规划页、行程详情页、我的行程页。
请优先设计对话规划页，采用左侧聊天、右侧行程卡片的布局。
```

### 8.4 AI 设计的边界

AI 适合：

- 出首版布局。
- 生成视觉方向。
- 补充图标、插画、文案草案。
- 产出组件代码骨架。

AI 不适合直接拍板：

- 复杂交互细节。
- 设计系统一致性。
- 品牌气质统一。
- 真实业务流程取舍。

## 9. MVP 页面优先级

第一优先级：

- 首页。
- 对话规划页。
- 行程详情页。

第二优先级：

- 我的行程页。
- 登录页。

第三优先级：

- 后台管理页。
- 分享落地页。

## 10. 推荐的首版设计产出物

建议你先做出以下设计物：

- 一张产品信息架构图。
- 一版桌面端首页线框图。
- 一版桌面端对话规划页线框图。
- 一版移动端对话规划页线框图。
- 一份组件清单。
- 一份颜色、间距、字体 token 草稿。

## 11. 首版交付标准

当 UI 满足以下条件时，可以进入开发阶段：

- 用户第一次进入能理解产品做什么。
- 用户可以在 10 秒内发起一次规划请求。
- 结果页不是纯文本，而是结构化可读。
- 用户可以对已有行程进行二次修改。
- 关键路径在桌面端和移动端都成立。
