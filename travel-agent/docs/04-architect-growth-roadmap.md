# Travel Agent 项目驱动的 AI Agent 架构师成长建议

## 1. 先说结论

你现在的起点其实很好：

- 熟悉 `LangChain`，说明你已经对 Agent 核心能力有感知。
- 做过 `FastAPI` demo，说明你对 Python 服务端并不陌生。
- 有前端背景，说明你更容易把产品体验、交互和工程落地串起来。

你当前最需要补的，不是“把所有技术都学一遍”，而是建立这三种能力：

- 把产品需求拆成系统模块的能力。
- 为每个模块做“够用技术选型”的能力。
- 把前端、后端、Agent、RAG、存储、部署串成一个可运行系统的能力。

如果你能围绕 `Travel Agent` 做到这些，你就已经从“前端程序员”走向“AI Agent 初级架构师”了。

## 2. 你要成为什么样的架构师

你现在不需要追求“高级架构师”。

当前目标应该是成为一个 **AI Agent 初级架构师**，也就是：

- 能把需求拆成前端、API、Agent、RAG、数据、部署几个部分。
- 能为每一层做轻量但合理的技术决策。
- 能做出 MVP，并解释为什么当前这样设计。
- 能说出未来如果业务增长，应该先升级哪一层。

这比“会很多技术名词”重要得多。

## 3. 学习原则：先会做，再会讲，再会选

很多人学不会架构，不是因为能力不够，而是顺序错了。

正确顺序是：

1. 先做出一个真实可运行的系统。
2. 再理解每一层为什么这样设计。
3. 再比较替代方案，形成自己的判断。

对于你来说，不建议先去系统刷完 `Next.js`、`Redis`、`PostgreSQL`、`Vercel` 的所有教程。

更建议用 `Travel Agent` 当主线项目，一边做，一边学。

## 4. 你对各项技术的掌握目标

### 4.1 Next.js

目标不是精通所有特性，而是达到“项目可用 + 架构可判断”：

- 能搭建 `apps/web` 和 `apps/admin` 两个应用。
- 理解 `App Router`、页面路由、布局、数据获取。
- 知道什么逻辑放客户端，什么逻辑放服务端。
- 能处理登录态、接口调用、页面拆分。

你真正要掌握的是：

- `app/` 目录结构
- `layout.tsx`
- `page.tsx`
- `loading.tsx`
- `route.ts`
- `fetch`
- 环境变量

### 4.2 Tailwind CSS

目标不是背类名，而是提升页面落地速度和设计一致性：

- 能快速搭用户端和管理后台页面。
- 能建立统一 spacing / radius / color token。
- 不写大量重复 CSS。

### 4.3 shadcn/ui

目标不是把所有组件都学一遍，而是把它当“可编辑组件基底”：

- 会用常见组件快速组装页面。
- 能修改组件样式与结构，而不是把它当黑盒。
- 能基于它抽共享组件包。

优先掌握这些组件即可：

- `Button`
- `Input`
- `Textarea`
- `Card`
- `Dialog`
- `Sheet`
- `Table`
- `Tabs`
- `DropdownMenu`
- `Toast`

### 4.4 FastAPI

你已经做过简单 demo，下一步是从“能返回数据”升级到“能承载产品接口”。

你要掌握：

- `Pydantic` schema
- 请求/响应模型
- 文件上传
- SSE 流式输出
- 鉴权中间件
- 错误处理
- 依赖注入

在这个项目里，至少要能做出：

- `POST /chat/stream`
- `POST /plans`
- `GET /plans/{id}`
- `POST /knowledge/upload`
- `GET /knowledge/list`
- `POST /knowledge/reindex`

### 4.5 SQLite

目标不是做 DBA，而是掌握业务建模与轻量存储。

你要能做到：

- 根据业务对象设计表结构。
- 理解主键、外键、索引。
- 会用 ORM + migration 管理数据结构。
- 知道 SQLite 什么时候够用，什么时候不够用。

这个项目建议先有这些核心表：

- `users`
- `sessions`
- `messages`
- `plans`
- `knowledge_documents`
- `knowledge_chunks`
- `tool_call_logs`
- `prompt_versions`

### 4.6 Chroma / 向量检索

目标不是研究所有向量数据库，而是先掌握一条完整可解释的 RAG 链路。

你要掌握：

- 文档切分
- embedding 生成
- top-k 检索
- metadata filter
- 引用来源返回
- 基本召回问题排查

在这个项目里，你需要能解释：

- 为什么召回了这几段内容。
- 是切分有问题、embedding 有问题，还是检索策略有问题。

### 4.7 Redis

现在不必强学。

你当前只需要知道 Redis 在下面这些场景里很有用：

- 限流
- 热点缓存
- 多实例共享状态
- 异步任务状态查询

建议策略是：

- 先把没有 Redis 的版本做完。
- 真遇到问题再引入 Redis。

这才是架构思维，而不是提前堆技术。

### 4.8 Vercel + Render

目标不是理解所有云平台细节，而是让项目真实上线。

你要掌握：

- 环境变量
- 构建命令
- 域名/回调地址
- 日志查看
- 持久化磁盘
- 健康检查
- 基本回滚

你的最小部署形态应该是：

- `Vercel` 部署 `apps/web` 和 `apps/admin`
- `Render/Railway/Fly.io` 部署 `FastAPI`
- `SQLite + Chroma` 放在持久化 volume

## 5. 你要建立的架构思维

### 5.1 从需求反推模块

一个旅游 Agent 产品，至少可以拆成：

- 用户前端
- 管理后台
- API 层
- Agent 编排层
- 工具层
- RAG 层
- 数据持久化层
- 部署与观测层

### 5.2 每层只解决自己的问题

- 前端：负责交互与展示
- API：负责协议、权限、聚合输出
- Agent：负责规划、调用工具、生成结果
- RAG：负责召回知识
- 数据库：负责状态持久化
- 部署：负责让系统稳定可运行

### 5.3 MVP 永远选最轻可行方案

当前项目就应该坚持：

- 数据库先 `SQLite`
- 向量检索先 `Chroma`
- 缓存先不引入 `Redis`
- Agent 先 `LangChain`
- 编排复杂后再升级 `LangGraph`

### 5.4 架构是演进出来的

你的目标不是“一步到位最优架构”，而是“当前最合适架构，并预留升级路径”。

比如：

- `SQLite -> PostgreSQL`
- `Chroma -> Qdrant`
- 单体 `FastAPI -> API + worker`
- 单 Agent -> 多节点编排

## 6. 你应该避免的坑

- 一开始就上最复杂架构。
- 前后端一起浅尝辄止，结果都不扎实。
- RAG 只会调通，不会评估召回质量。
- 不做数据建模，什么都塞 JSON。
- 项目还没稳定就拆微服务。
- 看很多教程，但没有成品。
- 把“会用工具”误当成“会做架构”。

## 7. 用这个项目完成进阶的建议路线

### 第 1 阶段：先成为能做出 AI 产品 MVP 的工程师

建议周期：2 到 4 周。

目标成果：

- `pnpm + workspace` monorepo 搭起来。
- `apps/web` 和 `apps/admin` 能跑。
- FastAPI 服务可用。
- 对话页面和知识库管理页面都能访问。

这阶段重点：

- `Next.js`
- `Tailwind`
- `shadcn/ui`
- `FastAPI`

### 第 2 阶段：成为能解释设计取舍的工程师

建议周期：2 到 3 周。

目标成果：

- 能解释为什么前端拆成 `web + admin`
- 能解释为什么当前先选 `SQLite`
- 能解释为什么 RAG 先选 `Chroma`
- 能解释为什么流式输出先用 `SSE`
- 能解释为什么当前不急着上 `Redis`

这阶段重点不是新增功能，而是把系统设计讲清楚。

### 第 3 阶段：成为知道什么时候该升级架构的人

建议周期：持续演进。

你要开始思考：

- 哪一层会最先成为瓶颈。
- 哪些问题出现后才值得引入 Redis。
- 什么情况下要把 SQLite 升级为 PostgreSQL。
- 什么情况下要把 Chroma 升级为 Qdrant。
- 什么情况下要从单 Agent 升级成显式状态编排。

做到这一步，你就已经具备初级架构师视角了。

## 8. 推荐你的项目实战顺序

### 第 1 周

- 建立 monorepo 骨架
- 建立 `apps/web`
- 建立 `apps/admin`
- 建立 `packages/ui`
- 建立 `packages/shared`
- 用户端完成聊天页
- 管理端完成知识库列表页

### 第 2 周

- 搭建 FastAPI API
- 接通 SSE 流式输出
- 用 SQLite 存会话/计划/知识库元数据
- 管理端完成文档上传接口

### 第 3 周

- 接入 Chroma
- 做文档切分、向量写入、检索
- 用户端对话接入知识库召回
- 返回引用来源

### 第 4 周

- 做基础部署
- 前端上 Vercel
- 后端上 Render / Railway / Fly.io
- 持久化 SQLite 和 Chroma
- 加入基础日志和错误监控

### 第 5 周

- 做第一次架构复盘
- 画系统设计图
- 画数据模型图
- 画调用链路图
- 补一份“为什么当前不引入 Redis / PostgreSQL / Qdrant”的设计说明

## 9. 每天应该怎么学

建议时间分配：

- 40% 做项目
- 30% 看官方文档
- 20% 复盘和画图
- 10% 看高质量最佳实践

顺序固定：

1. 先做
2. 遇到问题再查
3. 做完后总结
4. 最后抽象成自己的方法

## 10. 判断自己是否真的掌握

以后不要只问自己“我会不会这个技术”，而要问：

- 我能不能用它解决项目里的一个真实问题？
- 我能不能解释为什么选它？
- 我能不能说出它的边界和升级路径？

如果这三个问题你都能答出来，说明你已经不是“只会跟着教程做 demo”了。

## 11. 作为导师，我会给你的最终建议

- 不要贪多。
- 不要频繁换栈。
- 不要一开始追求最复杂方案。
- 不要先学抽象，再做项目。
- 先把 `Travel Agent` 做成自己的代表作。

你现在最合适的路线就是：

- 用 `Next.js + FastAPI + SQLite + Chroma + LangChain`
- 做出一个完整可用的 AI Agent 产品
- 在过程中把每一层理解到“能设计、能实现、能解释”

这条路线最稳，也最适合你从前端程序员成长为 AI Agent 初级架构师。
