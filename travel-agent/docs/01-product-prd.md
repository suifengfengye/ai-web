# Travel Agent 产品文档

## 1. 产品定位

Travel Agent 是一个面向中文用户的 AI 旅游助手，核心价值是把“旅游问答”升级为“可执行的出行规划”。

相较于通用聊天机器人，本产品聚焦以下场景：

- 用户输入目的地、时间、人数、预算、偏好后，自动生成可落地的行程方案。
- 结合实时天气、目的地知识库、公共资料，为用户提供更贴近当前场景的建议。
- 后续扩展机票、酒店、地图、签证、预算、收藏、导出等能力，逐步从 Demo 演进为完整产品。

当前 `agent-video/13th-knowledge-base-mcp` 已具备一个不错的原型基础，但仍属于“单脚本 Agent Demo”，适合验证思路，不适合直接作为独立项目长期演进。

## 2. 当前 Demo 能力评估

现有能力主要由一个 LangChain Agent 加 3 类工具构成：

- `search_wikipedia`：用于补充公开百科类知识。
- `juhe_mcp`：已接入聚合数据类 MCP，当前可扩展天气类能力。
- `knowledge_base`：本地知识库 MCP，适合沉淀自有旅游内容。

### 2.1 这 3 个 MCP / 工具是否足够

结论：足够做 Demo，不足以支撑正式旅游助手产品。

适合 Demo 的原因：

- 已经能覆盖“知识检索 + 天气 + 通用补充信息”三类最基础能力。
- 可以完成“给我一份某地 3 天攻略”这类演示型请求。
- 技术链路清晰，便于继续扩展。

不足的原因：

- 缺少地图 / 路线 / POI 营业时间，无法做真实出行决策。
- 缺少机票 / 火车 / 酒店 / 门票数据，无法做预订前决策。
- 缺少用户画像 / 偏好记忆，结果无法持续个性化。
- 缺少价格、预算、交通时效等结构化能力，难以稳定比较方案。
- 缺少安全、风控、数据质量与观测能力，不适合直接商用。

### 2.2 建议补充的 MCP / 工具能力

建议按阶段扩充，而不是一口气堆很多能力。

MVP 阶段建议补充：

- 地图与地理编码 MCP：城市、景点、经纬度、距离、路线、通勤时间。
- POI 信息 MCP：景点开放时间、评分、建议停留时长、门票参考。
- 酒店 / 住宿 MCP：区域推荐、价格带、通勤便利度。
- 交通 MCP：高铁 / 航班 / 城市内交通建议。
- 预算计算工具：根据天数、人数、住宿档次、交通方式估算预算。

成长阶段建议补充：

- 签证 / 政策 / 入境要求 MCP。
- 用户记忆工具：保存偏好、常用出发地、旅行节奏、忌口、亲子偏好。
- 行程导出工具：导出 Markdown、PDF、日历、地图收藏。
- 内容审核 / 风险提示工具：对高风险建议、过时信息做校验。

商用阶段建议补充：

- 订单系统 / 支付系统。
- 用户中心 / 历史会话 / 收藏夹。
- 多代理协作：行程规划 Agent、酒店推荐 Agent、交通 Agent、预算 Agent。

## 3. Prompt 策略

### 3.1 Prompt 是否应该固定

结论：不建议完全固定，也不建议完全放开。

建议采用“三层 Prompt 结构”：

- 系统层：固定，定义产品边界、工具调用规则、安全规则、输出质量要求。
- 场景层：半固定，针对“自由行攻略”“亲子游”“周末短途”“商务差旅”等场景切换模板。
- 用户层：开放，允许用户输入个性化要求，例如预算、偏好、节奏、同伴结构、餐饮限制等。

### 3.2 为什么不能完全固定

- 旅游场景天然个性化，固定 Prompt 会导致结果过于僵硬。
- 用户经常会补充偏好，例如“少走路”“带老人”“尽量地铁直达”“不吃辣”等。
- 独立产品需要支持多种交互方式，如问答、改写、重排行程、预算压缩、酒店替换。

### 3.3 为什么不能完全放开

- 如果不约束工具策略，模型可能跳过关键工具调用。
- 如果不约束输出结构，前端很难渲染为稳定 UI。
- 如果不约束安全边界，可能给出错误、过期或高风险建议。

### 3.4 推荐 Prompt 组织方式

建议拆为以下几部分：

- Base System Prompt：产品角色、安全要求、工具优先级、禁止事项。
- Planner Prompt：把用户需求转换为结构化旅行意图。
- Tool Policy Prompt：规定何时必须调用天气、知识库、地图、POI 等工具。
- Output Format Prompt：约束为结构化 JSON 或稳定 Markdown。
- Rewrite Prompt：在用户说“更省钱”“换亲子玩法”“压缩成 2 天”时用于增量重写。

## 4. 独立项目拆分方案

当前 Demo 需要从“教学脚本”升级为“可部署项目”，建议拆成如下结构：

```text
travel-agent/
  apps/
    web/                 # 前端 Web 应用
    api/                 # 对外 API 服务
    worker/              # 异步任务、索引、离线处理
  services/
    agent-service/       # LangChain / Agent 编排
    mcp-gateway/         # 统一 MCP 接入层
    rag-service/         # 知识库检索、重排、索引
  packages/
    shared-schema/       # 前后端共享类型
    prompt-lib/          # Prompt 模板与策略
    ui-tokens/           # 设计 token
  docs/
```

### 4.1 核心拆分原则

- Agent 编排与 Web/API 解耦。
- MCP 工具接入统一走网关或工具注册层，不散落在业务代码中。
- Prompt、Schema、工具策略单独管理，便于版本化。
- 输出尽量结构化，避免前端直接解析大段自由文本。

## 5. 产品范围建议

### 5.1 MVP 范围

建议先做成“旅游规划助手”，而不是“全流程 OTA”。

MVP 包含：

- 目的地攻略生成。
- 多轮追问与行程改写。
- 天气与知识库增强。
- 预算区间估算。
- 行程收藏与导出。

MVP 不建议一开始就做：

- 支付闭环。
- 实时库存预订。
- 复杂会员体系。
- 多语言国际化全量支持。

### 5.2 典型用户

- 周末短途游用户。
- 节假日自由行用户。
- 带老人 / 亲子出行用户。
- 想快速获得“可执行计划”的年轻用户。

## 6. 核心用户流程

### 6.1 首次规划

1. 用户输入目的地、日期、人数、预算、偏好。
2. 系统解析需求并补齐缺失参数。
3. Agent 调用知识库、天气、百科、地图等工具。
4. 输出结构化旅游方案。
5. 用户进一步要求“更省钱”“少走路”“加夜景”“换酒店区域”。
6. 系统增量改写，并支持保存版本。

### 6.2 行程管理

1. 用户保存攻略。
2. 用户导出 PDF / 图片 / 分享链接。
3. 出行前查看天气变化与注意事项。
4. 后续可进入“我的旅行”持续修改。

## 7. 服务端架构设计建议

### 7.1 总体架构

推荐采用“前端 Web + Python API + Agent Service + MCP / RAG 服务”的分层架构。

```text
Browser
  -> Web Frontend
  -> API Gateway / Backend
  -> Agent Orchestrator (LangChain Python)
  -> MCP Tools / RAG / External APIs
  -> Database / Cache / Object Storage
```

### 7.2 各层职责

- Web Frontend：承载对话、表单、结果渲染、收藏、登录。
- Backend API：用户鉴权、会话管理、接口聚合、限流、审计。
- Agent Service：Planner、工具调用、状态编排、输出结构化。
- MCP Gateway：统一接入第三方能力，隔离供应商差异。
- RAG Service：向量检索、重排、知识库管理、内容更新。

### 7.3 为什么不要直接把脚本暴露给外部

- 当前 `agent.py` 主要面向本地运行，不具备正式 API 服务能力。
- 缺少鉴权、限流、错误恢复、超时控制、日志追踪。
- 直接暴露会把 Prompt、工具细节和敏感配置耦合在一起。

## 8. LangChain Python 如何开放 API 供外部访问

建议使用 FastAPI 包一层 API 服务，而不是让外部直接调用脚本。

推荐接口：

- `POST /api/v1/chat`：普通问答。
- `POST /api/v1/plan`：创建一次新的旅游规划。
- `POST /api/v1/plan/{id}/revise`：增量修改行程。
- `GET /api/v1/plan/{id}`：获取已保存行程。
- `GET /api/v1/stream/{session_id}`：SSE 流式输出。

后端处理流程：

1. 接收前端请求。
2. 做参数校验与用户鉴权。
3. 组装结构化输入给 Agent。
4. 通过流式接口返回中间状态和最终结果。
5. 落库保存会话、方案、工具调用日志。

## 9. 部署与平台建议

### 9.1 Vercel 部署建议

Vercel 更适合部署前端和轻量 BFF，不建议直接承载重型 Python Agent 主服务。

推荐方式：

- Vercel：部署 Next.js 前端。
- Python API / Agent Service：部署到 Railway、Render、Fly.io、云服务器或容器平台。
- 数据库：使用 Supabase / Neon / PlanetScale / PostgreSQL。
- 向量库：使用 pgvector、Qdrant、Pinecone 之一。

### 9.2 原因

- Python LangChain 服务通常涉及长连接、流式输出、网络工具调用、RAG 检索。
- Vercel Serverless 对超时、冷启动、长任务不够友好。
- 前后端分离更便于稳定性治理和弹性扩容。

## 10. 其他容易遗漏的点

除了你已提到的产品、后端、Web 页面外，还需要提前考虑：

- 用户登录与权限。
- 会话历史与版本管理。
- Prompt 与工具调用的可观测性。
- 供应商切换能力，避免绑死单一模型或单一第三方 API。
- 成本控制，包括 token、外部 API、向量检索。
- 内容时效性，旅游信息容易过期。
- 安全与合规，避免输出错误政策、危险路线、误导性价格。
- 埋点与评估，明确什么叫“规划质量好”。
- 故障兜底，当工具失败时如何降级到通用回答。

## 11. 阶段性落地路线图

### Phase 1：原型升级

- 把现有 Demo 拆成前端、API、Agent 三层。
- 输出结构从纯文本升级为结构化 JSON。
- 补齐地图、POI、预算工具。

### Phase 2：MVP 上线

- 做 Web 页面对话与行程结果页。
- 支持登录、保存、分享、导出。
- 增加用户偏好记忆和行程改写。

### Phase 3：产品化

- 引入更完整的酒店 / 交通 / 门票能力。
- 增加观测、评估、风控和运营后台。
- 开始做增长、转化和商业闭环。
